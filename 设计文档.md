# 局域网剪切板与文件互传 —— 详细设计文档

> 版本：v0.3（事件驱动 + 严格防回环）
> 状态：设计阶段
> 关联文档：`文档.md`

---

## 1. 项目概述

一款 **自用** 的局域网剪切板 / 文件互传工具。核心场景：

> 在 A 电脑上 `Cmd+C` 复制 → B 电脑剪切板自动写入 → 在 B 电脑直接 `Cmd+V` 粘贴。

附带：选择文件/文件夹通过同款应用快速发到另一台机器。

技术栈：

- UI：**gpui + gpui-component**（Rust 桌面 GUI）
- 异步：**tokio**
- 传输：**WebSocket**（`tokio-tungstenite`），简单、跨平台、未来 Flutter 也好接
- 发现：**mDNS**（`mdns-sd`），通过 **独有服务类型** `_lanclip._tcp.local.` 只识别同款软件

**明确放弃**（自用场景下不需要）：加密、配对、PIN、TLS、密钥管理、断点续传、限速、i18n、企业网络兼容、SHA-256 校验。后续如确有需要再加。

## 2. 目标与非目标

### 2.1 目标 (In Scope)

- **零配置发现**：装了同款软件的设备在同一局域网内自动出现在列表。
- **剪切板自动同步**：
  - A 复制 → 自动推送 → B 写入系统剪切板 → B 直接粘贴。
  - 支持 **文本** 与 **图片（PNG）**。
  - 防回环（接收写入后不再当成"新复制"回推）。
- **手动文件发送**：选中设备 → 选文件/文件夹 → 发送 → 对方收到通知后保存到默认目录。
- **跨平台桌面**：macOS / Windows / Linux 一份代码。
- **核心与 UI 解耦**：协议层不依赖 gpui，未来 Flutter 安卓端可复用。

### 2.2 非目标 (Out of Scope)

- 加密 / 认证 / 配对（自用、可信局域网）。
- 跨公网传输、中继、账户、云同步。
- 断点续传、文件完整性强校验。
- iOS 端（剪切板权限受限）。
- 富文本、HTML、RTF 等复杂剪切板类型。

## 3. 技术选型

| 分层 | 选型 | 说明 |
|---|---|---|
| GUI | `gpui` + `gpui-component` | 自带跨平台渲染，性能好。 |
| 异步运行时 | `tokio` (multi-thread) | 网络/IO/定时任务。 |

| 工具 | `uuid`、`bytes`、`base64`、`image` | 按需引入。 |

## 4. 整体架构

### 4.1 分层

```
┌──────────────────────────────────────────────────┐
│  Presentation 层 (gpui)                          │
│   - Views / Components / 状态订阅                 │
└──────────────▲───────────────────────────────────┘
               │ (事件 / 命令, channel)
┌──────────────┴───────────────────────────────────┐
│  Application 层 (服务编排, 跑在 tokio runtime)    │
│   - DeviceService                                │
│   - ClipboardService                             │
│   - TransferService                              │
└──────────────▲───────────────────────────────────┘
               │
┌──────────────┴───────────────────────────────────┐
│  Domain 层 (纯模型)                               │
│   - Device / Peer / Message / TransferTask       │
└──────────────▲───────────────────────────────────┘
               │
┌──────────────┴───────────────────────────────────┐
│  Infrastructure 层                                │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌─────┐ │
│  │ network  │ │discovery │ │clipboard │ │store│ │
│  │  (ws)    │ │ (mdns)   │ │(clip-rs) │ │(fs) │ │
│  └──────────┘ └──────────┘ └──────────┘ └─────┘ │
└──────────────────────────────────────────────────┘
```

**关键约束**：

- UI 层只通过 **事件流 + 命令通道** 与 Application 层交互，不直接持有 tokio 句柄或网络对象。
- Application 层跑在 tokio runtime，通过 `tokio::sync::mpsc / watch / broadcast` 与 UI 线程通讯。
- 替换 UI（gpui → flutter）只需保留 Application 以下的层，未来通过 `flutter_rust_bridge` 暴露 API。

### 4.2 进程模型

单进程多任务：

- **UI 主线程**：gpui 事件循环，只做渲染与用户输入。
- **tokio runtime**：后台多线程，承载 mDNS、WebSocket 服务/客户端、剪切板轮询、文件 IO。
- 两者通过 channel 通讯，**禁止在 UI 线程做阻塞 IO，禁止跨线程持有 UI 句柄**。

### 4.3 模块依赖

```
app (binary, gpui)
 └── ui                  // gpui views
 └── application         // services
      └── domain
      └── infra::network    (tokio-tungstenite)
      └── infra::discovery  (mdns-sd)
      └── infra::clipboard  (clipboard-rs, 事件驱动)
      └── infra::storage    (fs + serde_json)
```

## 5. 核心模块设计

### 5.1 Domain 模型

```rust
pub struct DeviceId(pub String);   // 启动时随机生成的 uuid，存配置文件持久化

pub struct Device {
    pub id: DeviceId,
    pub name: String,              // 默认 = 主机名，可在设置中改
    pub os: OsKind,                // Mac / Win / Linux
}

pub struct Peer {
    pub device: Device,
    pub addr: SocketAddr,          // 由 mDNS 解析
    pub status: PeerStatus,        // Online / Offline / Connected
    pub last_seen: Instant,
}

pub enum ClipboardPayload {
    Text(String),
    ImagePng { width: u32, height: u32, data: Bytes },
}

pub struct TransferTask {
    pub id: Uuid,
    pub peer: DeviceId,
    pub direction: Direction,      // Send / Recv
    pub entries: Vec<FileEntry>,   // { rel_path, size }
    pub total_bytes: u64,
    pub state: TransferState,      // Pending/Running/Done/Failed/Cancelled
    pub progress: TransferProgress,// bytes_done, speed
}
```

### 5.2 Discovery（设备发现）

- **协议**：mDNS / DNS-SD。
- **服务类型**：`_lanclip._tcp.local.`（独有 → 只有装了本软件的设备会注册/响应，天然过滤）。
- **实例名**：`<device_name>._lanclip._tcp.local.`
- **TXT 字段**（明文即可，没有安全要求）：
  - `id=<uuid>` —— 用于稳定标识设备（设备改名后仍能识别同一台）
  - `name=<utf8>` —— 显示名
  - `os=mac|win|linux`
  - `v=1`（协议版本）
  - `port=<ws_port>` —— WebSocket 监听端口
- **行为**：
  - 启动时注册自身 + 持续 browse。
  - 维护 `PeerRegistry: HashMap<DeviceId, Peer>`，TTL（如 30s 未刷新）标记 Offline。
  - 通过 `broadcast::Sender<DiscoveryEvent>` 推送 `PeerAdded / PeerUpdated / PeerLost`。

### 5.3 Network（WebSocket）

**为什么 WebSocket**：单连接、消息边界天然、文本/二进制都支持、库成熟、Flutter 端用 `web_socket_channel` 也方便接。

#### 5.3.1 拓扑（1 条控制连接 + N 条数据连接）

- 每台设备同时是 **WS 服务端** 和 **WS 客户端**。
- 服务端：`tokio-tungstenite::accept_async`，监听随机可用端口（端口写入 mDNS TXT）。
- 客户端：当 mDNS 发现新 peer 时，按需建立连接。
- **连接结构**（per peer）：
  - **1 条控制连接**（`/lanclip/control`）：长存，承载 Hello/Ping/Pong/Clipboard*/Transfer*Offer/Accept/Done/Cancel。
  - **N 条数据连接**（`/lanclip/data`）：按需懒加载，**并发**传输文件。`N` 默认 4，可配置（`config.transfer_parallelism`）。
- **避免重复建连**：控制连接用 `DeviceId` 比较，大者主动连、小者只接受；数据连接由发送方按需主动连。
- **连接池**：
  ```rust
  struct PeerConnections {
      control: WsConnHandle,
      data: Vec<WsConnHandle>,   // 懒初始化，长度 ≤ N
  }
  type ConnectionPool = HashMap<DeviceId, PeerConnections>;
  ```

#### 5.3.2 各连接的职责

| 连接类型 | 帧类型 | 用途 |
|---|---|---|
| 控制 | `Message::Text(json)` | 元信息、剪切板内容、传输元数据与控制命令 |
| 数据 | `Message::Text(json)` + `Message::Binary` | `FileBegin` / 文件二进制流 / `FileEnd` |
| 心跳 | WebSocket `Ping/Pong` | 间隔 15s，60s 无 Pong 视为断开 |

剪切板永远走 **控制连接**（小数据、低延迟），不与文件传输互相阻塞。

#### 5.3.3 路径与子协议

- 控制：`ws://<host>:<port>/lanclip/control`
- 数据：`ws://<host>:<port>/lanclip/data`
- 子协议：`Sec-WebSocket-Protocol: lanclip.v1`

#### 5.3.4 握手

每条 WS 连接（无论控制/数据）建立后立即发送：

```rust
Hello {
    version: u16,
    role: ConnRole,          // Control | Data
    device: DevicePublic,    // id, name, os
}
```

对端回 `Hello`。控制连接握手后把 `Device` 写入 `PeerRegistry`，状态置 `Connected`；数据连接握手后挂到 `PeerConnections.data` 待 `TransferScheduler` 调度。

### 5.4 Clipboard（事件驱动 + 严格防回环）

#### 5.4.1 监听（不轮询）

使用 `clipboard-rs` 的 `ClipboardWatcher`：内部按平台用 **原生事件 API**

| 平台 | 机制 | 性质 |
|---|---|---|
| Windows | `AddClipboardFormatListener` + `WM_CLIPBOARDUPDATE` | 真·事件 |
| Linux X11 | `XFixesSelectionNotify` | 真·事件 |
| Linux Wayland | `wlr-data-control-unstable-v1`（KDE/Sway 等支持） | 真·事件 |
| macOS | `NSPasteboard.changeCount` 监听 | 系统层面没有真正可靠的剪切板事件；改用 changeCount 增量检测（仅一次轻量系统调用，**不读取 payload**）。这是 macOS 平台限制，无法避免。 |

监听器跑在独立 tokio task，触发后通过 `mpsc::Sender<ClipboardLocalEvent>` 投递给 `ClipboardService`。

#### 5.4.2 严格防回环算法

**问题**：A 复制 → B 写入本机剪切板 → B 的监听器触发 → 如果不加判断，B 会把刚收到的内容当成"新复制"再广播回 A → A 写入 → A 触发 → 无限循环。

**核心思路**：用 **内容哈希** 而非时间窗口去重；写入系统剪切板前 **先预登记 hash**，监听器命中时跳过广播。

```rust
struct ClipboardService {
    /// 本机已知的"当前剪切板内容" hash（无论来源是本地复制还是远端写入）。
    /// 监听器触发时，若新内容 hash == current_hash，说明这次变化是我们自己
    /// 刚写进去的（或与上次相同），不广播。
    current_hash: Arc<Mutex<Option<Hash>>>,

    /// 出口通道：本机真实复制 → 广播到所有 peer
    tx_broadcast: mpsc::Sender<ClipboardUpdate>,
}
```

**本机剪切板变化时**：
```
on_local_change(new_payload):
    new_hash = blake3(new_payload)
    let prev = current_hash.lock().take_or(None)
    if Some(new_hash) == prev:
        return                       // 与已知内容相同 → 跳过（含自写入回灌）
    current_hash = Some(new_hash)
    broadcast(ClipboardUpdate {
        origin: self.device_id,
        content_hash: new_hash,
        payload: new_payload,
    })
```

**收到远端剪切板时**：
```
on_remote_update(msg):
    if msg.origin == self.device_id:
        return                       // 兜底：不可能但防御
    // ★ 关键：先登记 hash，再写入系统剪切板
    current_hash = Some(msg.content_hash)
    write_system_clipboard(msg.payload)
    // 此刻系统会触发监听器 → on_local_change 计算出相同 hash → 跳过广播
    // 绝不把这条消息再转发给任何 peer（包括来源以外的 peer）
```

**不变量保证**：

1. 只有 `origin == self` 的事件才被广播；**收到的事件永不转发**（杜绝多设备场景下的扩散循环）。
2. 写入系统剪切板前 **必须** 先更新 `current_hash`；写入和登记之间需是临界区（`Mutex` 保护），否则可能监听器先触发再登记，导致漏抑制。
3. 平台监听器在某些情况下可能**先于**写入完成就触发（少见），用 hash 比对而不是"次数计数"才能可靠。
4. 用户连续两次复制同一内容（hash 相同）会被去重 —— 这是可接受的代价（自用场景几乎没影响）。

#### 5.4.3 类型与大小

- **类型优先级**：image > text；同一时刻只发一种。
- **大小限制**：
  - 文本：1 MiB
  - 图片：8 MiB（超过本次不同步，UI 提示"过大，改用文件发送"）
- 大小检查后再算 hash，避免对超大数据浪费 CPU。

#### 5.4.4 广播策略

- 本机真实复制 → **向所有已连接 peer 广播**（fan-out）。
- 每条 `ClipboardUpdate` 都带 `origin = self.device_id` 与 `content_hash`，便于接收端校验/调试。

### 5.5 Transfer（文件传输 —— 多连接并发）

#### 5.5.1 并发模型

- 同一对设备之间，**N 条数据 WS 连接 = N 个并发 worker**（默认 `N = 4`，跟随 CPU 核数上下浮动可配置）。
- **粒度 = 单文件**：每个 worker 一次独占处理一个 entry（不切分单文件），传完取下一个。
  - 实现简单（无需 chunk offset 重组）。
  - 多文件场景下 N 路并行，吞吐 ≈ N × 单流；单巨大文件场景退化为单流（接受 —— 千兆下单流已能跑 ~100 MB/s）。
- worker 是独立 tokio task，分布在 tokio multi-thread runtime 的不同 worker thread 上 →  天然多线程。
- 文件读取用 `tokio::fs::File` + `BufReader`，写入用 `tokio::fs::File`；IO 不阻塞 runtime。

#### 5.5.2 发送方流程

1. **准备**：用户在 UI 选目标 peer + 文件/文件夹。`TransferService` 遍历目录构建 `entries: Vec<{ entry_id, rel_path, size }>`（`entry_id` 单调递增），算 `total_bytes`，生成 `task_id = Uuid::new_v4()`。
2. **协商**（控制连接）：发 `TransferOffer { task_id, entries, total }`，等待 `TransferAccept { task_id }`。
3. **建数据连接**：检查 `PeerConnections.data`，按需建到 N 条。
4. **调度**：`TransferScheduler` 把 entries 推入 `mpsc::Sender<EntryJob>` 队列；N 个 worker 各持 `mpsc::Receiver` 的 `Arc<Mutex<…>>`（或用 `async-channel` 多消费者）抢任务。
5. **worker 内单文件传输**（数据连接）：
   1. 发 `FileBegin { task_id, entry_id, rel_path, size }`（JSON Text）。
   2. 流式读取，按 64 KiB chunk 发 `Message::Binary(bytes)`。
   3. 发 `FileEnd { task_id, entry_id }`（JSON Text）。
   4. 期间向控制连接的 `ProgressTracker` 报告 `bytes_done`（用 `mpsc` 投递，不阻塞 worker）。
6. **完成**：所有 entries done → 控制连接发 `TransferDone { task_id }`。

#### 5.5.3 接收方流程

- **控制连接**：收到 `TransferOffer` → 自动接受（自用免确认；可在设置中改为弹窗）→ 回 `TransferAccept`，并在内存中创建 `RecvTaskState { task_id, expected: HashMap<entry_id, FileEntryMeta>, done: HashSet<entry_id> }`。
- **每条数据连接**：独立消费 `FileBegin → Binary…N → FileEnd`：
  1. 收 `FileBegin` → 校验 `task_id, entry_id` 存在；按 `<download_dir>/lanclip/<sender_name>/<rel_path>` 规范化路径（剔除 `..` / 绝对路径前缀），创建/截断 `*.lanclip.part` 临时文件，记录 `current_entry` 状态。
  2. 收 `Binary` → 追加写入 `current_entry` 文件句柄。
  3. 收 `FileEnd` → 关闭句柄，`rename` 去掉 `.part` 后缀，在 `RecvTaskState.done` 中标记 `entry_id`，向控制连接的 `ProgressTracker` 上报。
- **整体完成**：收到控制连接的 `TransferDone` → 校验 `done == expected_set`，匹配则发 UI 通知；不匹配则回 `TransferFailed`。

#### 5.5.4 状态机

```
TransferTask:
  Pending          // 已 Offer，等 Accept
    → Accepted     // 收到 Accept，开始建数据连接
    → Running      // 任意 entry 开始传输
    → Done         // TransferDone 双向确认
    → Failed       // 任一 worker 报错（不可恢复）
    → Cancelled    // 用户/对端发 TransferCancel
```

#### 5.5.5 取消

- 用户在 UI 点取消 → 控制连接发 `TransferCancel { task_id }`。
- 双方所有 worker 检查到 `task.state == Cancelled` → 当前文件完结后停止取下一个 entry；当前文件可直接断开数据连接以快速终止（连接断开不影响其他 task）。
- 接收端清理 `*.lanclip.part`。

#### 5.5.6 不做的事（v1）

- 不做 SHA-256 校验（WS 基于 TCP 已经有完整性，自用足够）。
- 不做断点续传（自用场景重传就行；如需，将来在 `TransferAccept` 里加 `resume: HashMap<entry_id, offset>`）。
- 不做限速（局域网内一般不需要）。
- 不做单文件切片并发（巨大单文件退化为单流；未来若有需求可在协议帧增加 `offset` 字段升级）。

## 6. 通信协议

### 6.1 控制消息（WebSocket Text Frame，JSON）

带 `type` 字段做 tagged union；用 `serde(tag = "type")` 派生：

```rust
#[derive(Serialize, Deserialize)]
#[serde(tag = "type", rename_all = "snake_case")]
pub enum Msg {
    // 每条连接建立后第一帧；role 区分控制/数据
    Hello       { version: u16, role: ConnRole, device: DevicePublic },
    Ping        { ts: u64 },
    Pong        { ts: u64 },

    // 剪切板（控制连接）：必带 origin + content_hash，便于严格防回环（详见 5.4.2）
    ClipboardText  { origin: String, content_hash: String, text: String },
    ClipboardImage { origin: String, content_hash: String, width: u32, height: u32, png_b64: String },

    // 文件传输元数据（控制连接）
    TransferOffer  { task_id: Uuid, entries: Vec<FileEntryMeta>, total: u64 },
    TransferAccept { task_id: Uuid },
    TransferReject { task_id: Uuid, reason: String },
    TransferDone   { task_id: Uuid },
    TransferCancel { task_id: Uuid },
    TransferProgress { task_id: Uuid, bytes_done: u64 },  // 可选，发送端 → 接收端 / 双向

    // 文件流锚定（数据连接）
    FileBegin   { task_id: Uuid, entry_id: u32, rel_path: String, size: u64 },
    FileEnd     { task_id: Uuid, entry_id: u32 },
}

#[derive(Serialize, Deserialize)]
#[serde(rename_all = "snake_case")]
pub enum ConnRole { Control, Data }

#[derive(Serialize, Deserialize)]
pub struct DevicePublic { pub id: String, pub name: String, pub os: String }

#[derive(Serialize, Deserialize)]
pub struct FileEntryMeta { pub entry_id: u32, pub rel_path: String, pub size: u64 }
```

### 6.2 文件数据（WebSocket Binary Frame，数据连接）

每条数据连接的消费规则：

- 状态机：`Idle → FileBegin → (Binary)* → FileEnd → Idle`
- Binary 帧本身 **不带头部**，归属由前一个 `FileBegin` 锚定；接收端按 `entry_id` 关联到对应文件句柄。
- 单条数据连接上 **同一时刻只有一个 entry 进行中**（顺序串行）；并发由 **多条数据连接** 实现。
- 单个 Binary 帧 ≤ 64 KiB（与 WS 默认帧大小友好）。
- 多条数据连接的 entry 顺序之间 **无保证**；接收端凭 `entry_id` 各自落盘，互不干扰。

### 6.3 版本协商

- `Hello.version` 取 `min(local, peer)`；若低于最低支持版本 → 断开 + UI 提示升级。

### 7.2 状态流

## 13. 风险与开放问题

- **gpui 生态成熟度**：托盘、自启动、拖拽接收文件等可能要调 OS API（`tray-icon`、`auto-launch`）或自己写平台代码。
- **macOS 剪切板事件**：系统未提供可靠的剪切板事件 API，`clipboard-rs` 在 macOS 上仍基于 `NSPasteboard.changeCount` 检测。这是 **不可避免的平台限制**，但 changeCount 只是一次轻量系统调用，不读取 payload，性能影响可忽略。
- **Linux Wayland 兼容性**：依赖 `wlr-data-control-unstable-v1` 协议，KDE/Sway/Hyprland 等支持；**GNOME 暂不支持** → 在 GNOME Wayland 下需降级为 X11 兼容层（XWayland）或提示用户。
- **写入-监听竞争**：写入系统剪切板与 `current_hash` 登记必须在同一临界区（详见 5.4.2 不变量），否则可能漏抑制导致回灌。需要写单元测试覆盖。
- **多网卡**：mDNS 默认在所有接口广播，一般够用；若发现失败，UI 提供"选择网卡"高级设置。
- **同款不同版本**：用 `Hello.version` 协商，低于支持版本直接拒连。
- **未加密带来的风险**：剪切板可能含敏感内容（密码、token）。**仅在可信局域网（家庭/个人）使用**；未来若上公司网络/酒店 Wi-Fi，需补加密层（可平滑加 `rustls` + 预共享密钥）。

## 14. 术语表

- **mDNS / DNS-SD**：组播 DNS / DNS 服务发现，用于零配置局域网发现。
- **WebSocket**：基于 TCP 的全双工消息协议，支持文本/二进制帧。
- **回环抑制**：避免接收 → 写本机剪切板 → 轮询又当作"新复制"广播出去，造成无限循环。
